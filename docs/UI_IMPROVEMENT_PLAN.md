## Redesigns (dedicated session each)

- **Render architecture: keyed node-reuse + rAF coalescing + per-slice diff gating** — scope: 2-3 dedicated sessions. Highest-regression-risk item in the plan — needs the panel-by-panel migration order, a side-by-side verification pass per panel against live 6Hz data, and the interaction-lock quick win landed first as safety net. Relay-only deploys throughout.
  Convert the hot panels (needs, bio, skills, gear, health, Orders modal) from innerHTML string rebuilds (48 sites) to build-once/mutate renderers holding refs to value nodes; handlePawnState writes to a latestState var and a single requestAnimationFrame loop renders newest-state-once-per-frame; each panel keeps a signature of its input slice and skips rendering when unchanged. Then split app.js (5791 lines) into ES modules (net/store/commands/panels/*) with a tiny subscribe store, no build step. Delete most of captureCommandFocus/restoreCommandFocus once nodes survive ticks natively. Do it panel-by-panel behind the existing renderers so nothing regresses mid-migration.
  Why: The dropdown deaths, Buy-search keystroke fights, and per-frame layout cost at 10+ concurrent viewers are all the same architectural fact: network cadence directly drives wholesale DOM destruction. The quick-win guard stops the bleeding; this removes the disease. A panel that doesn't re-render can't kill its dropdown.

- **Command lifecycle: sent → acknowledged → applied, with Buy never optimistic** — scope: 1-2 sessions spanning all three layers: mod (ack/error echo + failure outcomes into ActionLog), relay (routing acks to the right client + admin), client (ledger + visual states). Protocol addition — needs a mod restart and backward compatibility for viewers on the old client.
  Give every command a client id and a pending ledger: on click the specific control shows 'sent' (muted pip, <200ms) as an overlay that never overwrites payload truth; 'acknowledged' on relay/mod echo (requires adding command_ack/command_error to the mod's protocol); 'applied' when a later pawn_state reflects the change; 5s timeout reverts to payload value with a 'didn't take — game may be paused' notice. Buy purchases render dashed-amber 'pending' until execution confirms, then the coin balance updates exactly once with a single 300ms pulse — never an optimistic balance. Surface per-viewer failure counts to the streamer (ActionLog aggregation + admin strip) so silent-failure viewers become visible.
  Why: This is the structural fix for today's core lesson: viewers couldn't tell payload-truth from UI-truth, fire-and-forget send() trains spam-clicking that floods the relay, and the streamer currently has ZERO visibility into a viewer failing every command for 10 minutes (lastCommandLabel is set pre-execution, PawnCommandRouter.cs:154-164). Financial state (real Toolkit coins) makes the no-optimism rule non-negotiable.

- **Gear inversion + authoritative option lists from the game** — scope: 1 session. Mod changes (payload additions) + relay passthrough + client renderers. Mod-restart deploy; payload additions must be backward-compatible so old clients don't break mid-stream.
  Invert the Gear tab: the flat 'Wearing now' list becomes the primary, guaranteed-complete display (every payload item with quality/HP); rebuild the slot rig to stack N items per body group (fixing the first-wins Map at app.js:2187 and the unreachable 'other-N' keys at :2350) or retire it. Adopt the panel-wide rule: no renderer may drop a payload item because the layout lacks a cell — overflow renders '+2 more', never silence. Same session: mod sends authoritative option lists (trait defName+degree as unique key, labels resolved game-side; same for schedule and work types) so dropdowns are built from game defs, never assembled client-side from observed pawn data.
  Why: The gear bug and the traits-dropdown bug are the same failure from opposite directions: client-side structure assumptions discarding or inventing data the game owns. Fixing both at the source ends the whack-a-mole; the client dedupe quick win is explicitly a stopgap.

- **Streamer surface consolidation: exception strip, shared tokens, feature parity** — scope: 1-2 sessions. admin.html restructure + tokens is relay-only; parity items need mod-side data (tickets, timeout status, permission commands over the admin WS) so full deploy is both. Split: session 1 relay-only strip/theme, session 2 parity protocol.
  Restructure admin.html around a fixed top exception strip answering 'does anything need me?' in one glance — pending-claims count, failed-command count, active vote, relay/game connection as large chips, quiet gray when fine, amber/red with text labels when not; Claim Requests auto-sorts to top; Broadcast/Vote/Log below the fold. Port the viewer's token block (bg/surface/border/text hierarchy, gold accent, square radius, IBM Plex) dropping parchment/wood textures — flat and dense. Close the parity gaps that force alt-tabbing on camera: permissions grid, ticket counts, revive, per-viewer last-activity age and banned/timeout status in the web admin; replace the Spawn prompt() with a viewer picker; separate Ban from the flat button bar with a red danger treatment. Fix the inert 'Default permissions' grid in the in-game empty inspector (OverlordMainTabWindow.cs:759-762) — wire it to real persisted defaults or render it read-only, never as live-looking checkboxes that discard clicks.
  Why: The streamer uses these surfaces while LIVE with 5-10 seconds of attention; the current equal-weight h2 stack requires a read-through, the generic blue-on-gray theme breaks product identity in every tutorial screenshot, and neither surface is sufficient alone so incidents mean alt-tabbing on camera. The inert grid is a control that lies.

- **Lobby as live spectator view + activation funnel** — scope: 1 session, almost entirely relay-only (lobby rendering reuses existing map/roster handlers; instrumentation is two relay timestamps). Care needed not to regress the claim flow — the claim-button re-entrant renderer bug (app.js:1871-1876) gets fixed as part of touching this screen.
  Render the JPEG map stream and colonist roster (startLobbyRefresh already fetches both every 2.5s) BEHIND the claim state on screen-lobby so waiting is a working demo; add the social-proof line to ops-header ('8 viewers in · 6 of 9 colonists claimed') with claimed pawns showing their controller's Twitch name; on assignment, surface the existing free appearance change as the suggested first action ('Make [pawn] yours — one free hairstyle change'); turn lobby-death-card from obituary into re-entry (days survived, kills, commands issued + 'Rejoin the queue'); instrument panel-open→claim and claim→first-accepted-command timestamps in the relay, targeting <3 min.
  Why: Viewers arrive cold from chat and currently land on 'Waiting for assignment' — an empty zero-state for a product whose colony data exists independent of the viewer. The endowment hook (free, visible on stream, social proof to chat) bonds viewer to pawn before the first paid Buy decision; the death card is the peak-end moment the whole session gets judged by; the funnel instrumentation tells us where cold viewers actually fall out instead of guessing.

## Do NOT do

- Do not keep expanding the captureCommandFocus/restoreCommandFocus whack-a-mole or add per-dropdown guards — the tower's position is explicit that this is an architecture decision, not a guard (Artist dashboard_design §7; Programmer javascript.md). Every new guard is debt against the keyed-renderer redesign.
- Do not add icons to the tab row, nav sections, or Buy rows, and do not add sentiment/mood labels to needs bars — the viewer panel is currently compliant with Tell #1/#2 and the audit cleared its text-only chrome; keep it that way. The command gizmo icons are deliberate RimWorld fidelity (style.css:1187 comment) and stay.
- Do not animate anything fed by the 6Hz stream — no CSS transitions on needs bars, no counting animations on coins, no number tweens. Instant textContent/width swaps + tabular-nums only; animation is reserved for slow-cadence events (data_dense_ui_patterns 'Number Transition Animations'). Transitions at 6Hz queue, lag, and are part of why the panel feels untrustworthy.
- Do not ever show an optimistic coin balance or a success state before the game confirms — financial state is never optimistic (cognitive_load_performance); pending purchases get the dashed-amber at-risk treatment, and the balance updates exactly once on confirmation.
- Do not add an eighth tab to the drawer — the Health/Skills/Gear/Social/Events/Log/Chat row is already at the ~7-chunk working-memory ceiling (Miller's Law, data_dense_ui_patterns). New features fold into existing tiers or replace something.
- Do not blank panes on websocket drop or show skeletons over live data — stale-while-revalidate with an honest freshness stamp, never empty regions; skeletons are for TRUE first load only and must be guarded from the 6Hz path (Marketing module 32 + today's exact ambiguity lesson).
- Do not restyle admin.html with the parchment/wood raster textures when porting tokens — it is a utility surface; port the palette, gold accent, square radius, and type only, keep it flat and dense (dark_mode_saas_cookbook accent-deep-not-palette-deep).
- Do not fix the traits dropdown long-term by smarter client-side label heuristics — option lists must come from the game's defs (defName+degree keys); client dedupe is a labeled stopgap only (constraints beat validation).
- Do not render controls that don't work — the inert default-permissions grid pattern (OverlordMainTabWindow.cs:759-762) must never be repeated; a control either functions or renders read-only/absent.
- Do not add new uppercase-label classes, glow shadows, gradient accents, or radius dialects while touching style.css — the surface already has 9 uppercase classes, a glowing phase-dot, and 4 radius values against a declared --radius:0 system; every touch should reduce that count, not grow it (Tells #3/#6 + de-slop 'earn every element').
- Do not replace the death card, empty states, or lobby copy with marketing prose, coaching lines, or icon empty-states while doing the lobby redesign — one quiet factual line per state (Tell #7/#8; the Buy tab's 'Waiting for Twitch Toolkit prices…' is the house-standard pattern).
- Do not batch the quick wins into one mega-commit or land the render redesign wholesale — the product is live nightly with 10+ viewers; one change at a time, panel-by-panel migration, verify against the live 6Hz stream before the next slice (Engineering Discipline §1 + real-user-walkthrough rule).
